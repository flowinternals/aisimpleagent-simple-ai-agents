## Gaps In Automated Coverage

Highest-value **non-live** gaps (MVP regression risk):

1. **No tests for `httpErrorFromProviderAdapter` / `mapGenerationErrorToHttp`** - adapter codes to HTTP status + client-safe messages are untested; UI recovery depends on stable codes (`LIVE_PROVIDER_QUOTA_EXCEEDED`, `ADAPTER_RESULT_INVALID`, etc.).
2. **No tests for `normalizeGenerationAdapterResult`** - malformed adapter payloads can become opaque 502s; contract is substantial and isolated.
3. **No tests for `generationAgentService.js`** - `buildProviderPrompt` (ROLE, theme, aspect ratio, USER SPECIFICATION delimiters) unprotected.
4. **No tests for `providerAdapter.js` / `mockResultProvider.js`** - mock branch and live dispatch switch untested without network.
5. **No in-process route tests** - 415/400/200 depend on running server (`test:readiness`) or manual fetch.
6. **No frontend tests** for `parseApiHealthResponse`, `liveHealthForProviderId`, or `generateDisabled` logic in `App.tsx`.
7. **No browser E2E** - form submit, settings persistence, download click, fullscreen preview, preview `onError`.

Live-specific gaps:

- Readiness script does not offer `--live` for Google (OpenAI only).
- Live image generation not executed in this review.

## Gaps Found

### Behavioural and UX

| Gap | Severity | Detail |
|---|---|---|
| Live Generate enabled while health unknown | Medium | `generateDisabled` only blocks when `selectedLiveHealth !== null && !ready`. While `apiHealth` is null (initial load) or after health fetch failure (`liveHealthError`, `apiHealth` null), Generate stays enabled and live calls can fail server-side instead of being blocked in UI. |
| Health fetch deps omit `providerId` | Low | `useEffect` deps: `[providerSettings.providerMode, settingsOpen]`. Both vendors are returned in one health payload, so target changes while settings are open still read correct `selectedLiveHealth` from cache; refetch on target-only change is unnecessary unless health shape becomes per-target lazy. |
| Mock mode after error | Low | `switchToMockMode` clears `previewError` but does not auto-retry; user must submit again (acceptable). |
| Failed generate clears stale preview | OK | `handleGenerate` sets `result` null at start; preview phase tests cover error-over-success. |

### Test and regression

- Punctuation-only, max length, strict keys, 415, `INVALID_JSON`: exercised via HTTP in this run but **not** in committed regression files (only partial unit coverage for validation).
- `generationAgentService`, `providerAdapter`, adapter normalization, HTTP error mapping: **zero** dedicated tests (grep confirms).
- Browser download/preview/settings: untested end-to-end.

### Architecture and consistency

- `IMPLEMENTED_LIVE_PROVIDER_IDS` duplicated in `server/validation/generationRequest.js` and `src/types/providerSettings.ts` (drift risk).
- Live theme is primarily prompt text in `buildProviderPrompt`; adapters receive `imageTheme` for mock palette only (OpenAI/Google use prompt + size/quality).

## Overall Assessment

The **mock path is production-ready** for MVP demo and local testing: unit coverage is broad on shared/UI helpers, `npm test` and `npm run build` pass, and `test:readiness` plus HTTP edge cases confirm the running API behaves correctly for validation and mock generation.

The **live path is implemented and health-aware** but less proven: this review did not run billed live generation. The largest MVP risks are (1) **untested error and result contracts** at the adapter boundary, and (2) **live Generate enabled before health is known**, which can produce confusing failures when live mode is actively used. Addressing **D2, C1, and C3** now gives the best regression return without expanding scope; treat **D1** as a live-mode-only follow-up.
